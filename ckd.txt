# CEMP Mobile: Android-First Post-Quantum Messaging on Nervos CKB

## 1. Product definition

CEMP Mobile is a direct-messaging application in which encrypted message transport is performed through Nervos CKB cells.

The application should resemble a conventional private messenger:

* Conversation list
* Contact profiles and avatars
* Text and image messages
* Delivery and acknowledgement states
* QR-based contact exchange
* Local notifications
* Searchable local message history

Unlike a conventional messenger, there is no central messaging server holding user messages. Encrypted payloads are temporarily published to CKB, downloaded by the recipient, decrypted locally, and then reclaimed by the sender after acknowledgement.

The blockchain is the transport and temporary availability layer. The phone is the permanent message store.

The initial target is Android. The architecture must keep platform-specific services isolated so that an iOS client can later share the protocol, cryptography, transaction-building, database schema, and most UI logic.

---

# 2. Core principles

## 2.1 Messaging application first

The user should interact with CEMP Mobile as a messaging application, not as a blockchain explorer or general-purpose wallet.

Blockchain details should normally be translated into familiar states:

| Blockchain state                      | User-facing state     |
| ------------------------------------- | --------------------- |
| Transaction being assembled           | Preparing             |
| Transaction submitted                 | Sending               |
| Transaction in mempool                | Pending               |
| Message cell committed                | Sent                  |
| Recipient response references message | Received              |
| Sender reclaims message cell          | Cleared               |
| Transaction rejected                  | Failed                |
| Insufficient CKB capacity             | Messaging balance low |

Transaction hashes, cell outpoints, fees, lock scripts, and capacity details belong in an optional advanced details screen.

## 2.2 Local-first storage

Permanent user data is stored only on the device unless the user explicitly exports a backup.

This includes:

* Contacts
* Contact names and notes
* Contact avatars
* Conversation history
* Decrypted messages
* Downloaded images
* Drafts
* Read state
* Transaction mappings
* Reclaim state
* Profile metadata
* Application preferences

On-chain data should be considered temporary encrypted transport data.

## 2.3 Sender-funded temporary storage

The sender provides the CKB capacity needed to create:

* Text message cells
* Image manifest cells
* CKBFS data cells
* Any protocol metadata cells

The sender retains authority to reclaim those cells.

Recipients should not have to spend their own CKB merely to receive a message. Sending a response is a new outbound operation funded from the recipient’s own messaging float.

## 2.4 Explicit wallet warning

The wallet is an operational wallet for CEMP Mobile.

The onboarding and wallet screens should state:

> This wallet is intended to hold a limited CKB balance for messaging operations. It has not been designed, audited, or recommended as a primary wallet or long-term store of funds.

The application should encourage a configurable maximum operational balance rather than encouraging users to deposit substantial funds.

# 3. Findings from the supplied projects

The supplied projects establish a useful prototype foundation:

* `cemp-pq` implements ML-KEM-768 encapsulation, AES-256-GCM payload encryption, ML-DSA-65 signing support and Molecule structures.
* It has testnet examples for profile creation and encrypted message publication.
* `ckb-mldsa-lock` supplies deployed and smoke-tested post-quantum lock variants.
* ML-DSA-65 currently produces an approximately 5.3 KB witness.
* The supplied ML-DSA deployment is explicitly not ready for mainnet.
* The readiness checklist still requires fuzzing, cryptographic vector testing, external review, reproducible builds and a complete protocol specification.
* The current benchmark documentation notes that the demonstrated signing digest covers the transaction hash rather than the complete production-grade CKB sighash-all construction.
* `key-vault-wasm` provides useful patterns for password-based encrypted seed storage, Scrypt, HKDF, IndexedDB and secure buffer handling, but it is designed around SPHINCS+ and a custom multi-part mnemonic format.

Consequently:

1. Initial development should target CKB testnet.
2. The application must not hard-code a single lock deployment into business logic.
3. The key vault needs an ML-DSA/ML-KEM-specific implementation.
4. Standard BIP39 compatibility should not reuse the custom 36/54/72-word SPHINCS+ mnemonic system.
5. A security review and upgraded transaction signing implementation are hard gates before mainnet.

# 4. Proposed system architecture

## 4.1 Recommended application stack

For an Android-first application that must later support iOS:

### Application layer

* React Native with TypeScript
* Native Android modules in Kotlin where required
* Native iOS modules in Swift later
* React Navigation
* TanStack Query or a small purpose-built state synchronisation layer
* Zustand or Redux Toolkit for transient application state

React Native is a practical fit because the supplied CEMP code and CCC integration are already TypeScript-based.

### Shared protocol package

Create a standalone TypeScript package:

```text
packages/cemp-core
```

It should contain:

* Protocol schemas
* Message serialization
* Payload encryption and decryption
* Transaction construction
* Cell discovery
* Reclaim transaction construction
* Receipt processing
* Conversation identifiers
* Network configuration
* CKBFS manifest handling
* Compatibility tests and vectors

It must not import React Native UI modules.

### Native security package

Create:

```text
packages/cemp-secure-vault
```

With a TypeScript-facing interface and native implementations:

* Android Keystore integration
* iOS Keychain/Secure Enclave-compatible wrapping later
* Encrypted vault file
* Password/PIN derivation
* Biometric unlock
* Memory clearing where practical
* Transaction signing without exporting secret keys to ordinary JavaScript

For the earliest testnet prototype, a WASM or TypeScript signer can be used behind the same interface, but production signing should migrate into audited Rust or native code.

### Local database

Use encrypted SQLite rather than AsyncStorage for core data.

Recommended logical components:

* SQLite
* SQLCipher or equivalent database encryption
* Attachment files encrypted separately
* Encryption key wrapped by Android Keystore
* Database migrations with explicit schema versions

# 5. Identity and wallet specification

## 5.1 One mnemonic, separated purposes

The wallet must use a standards-compliant BIP39 mnemonic:

* 12 or 24 English words
* Valid BIP39 checksum
* Optional BIP39 passphrase
* Full import/export compatibility at the mnemonic level

The application should derive one root seed using the standard BIP39 seed function and then derive independent domain-separated sub-seeds.

Example:

```text
BIP39 seed
 ├── CEMP/CKB/ML-DSA/identity/v1
 ├── CEMP/CKB/ML-KEM/messaging/v1
 ├── CEMP/LOCAL/database/v1
 ├── CEMP/CONTACT/exchange/v1
 └── CEMP/BACKUP/encryption/v1
```

Use HKDF-SHA-256 or HKDF-BLAKE2b with explicit domain strings.

Do not use non-hardened BIP32 derivation for post-quantum private keys. BIP39 is the recovery container; domain-separated HKDF is the deterministic key derivation mechanism.

## 5.2 Generated key material

Initial identity:

* ML-DSA-65 signing key pair
* ML-KEM-768 encryption key pair
* Optional classical compatibility key only where necessary for tooling or migration
* Random local database key
* Random attachment encryption key

The ML-DSA and ML-KEM seeds must be independently derived. They must not reuse the same bytes directly.

## 5.3 On-chain profile

Each user may create one discoverable CEMP profile cell containing:

* Protocol version
* ML-DSA algorithm and parameter identifiers
* ML-DSA public key
* ML-KEM algorithm and parameter identifiers
* ML-KEM public key
* Messaging address or lock script hash
* Supported protocol versions
* Supported attachment formats
* Optional display handle
* Optional profile icon hash or small icon reference
* Key creation epoch
* Key rotation sequence
* Previous profile reference when rotated
* Revocation state

Sensitive contact information must not be placed in this profile.

The stable identity should be the profile Type ID rather than a mutable display name.

## 5.4 Address display

The app should display:

* Human-readable CKB address
* CEMP Profile ID
* QR code containing a versioned contact bundle

Contact bundle example:

```json
{
  "protocol": "cemp-contact",
  "version": 1,
  "network": "ckb_testnet",
  "profileTypeId": "0x...",
  "lockScriptHash": "0x...",
  "address": "ckt1...",
  "fingerprint": "ABCD-1234-..."
}
```

The QR should not contain secret material.

## 5.5 Wallet functionality

Required wallet operations:

* Create wallet
* Import BIP39 mnemonic
* Export mnemonic after authentication
* Display CKB address
* Display confirmed balance
* Display pending balance
* Receive CKB
* Send CKB
* Scan payment QR
* Display transaction history
* Estimate fees
* Warn about low messaging capacity
* Reserve capacity for pending messages
* Reclaim eligible message and image cells
* Consolidate fragmented cells
* Change RPC and indexer endpoints
* Switch between testnet and supported future networks

The wallet should distinguish:

```text
Total balance
Available balance
Reserved for pending messages
Reclaimable capacity
Pending transaction capacity
```

# 6. Revised CEMP message cell model

The prototype’s separate recipient-owned notification cell conflicts with prompt sender reclamation. If a cell is locked to the recipient, the sender cannot directly reclaim its capacity.

The production design should instead use sender-owned cells with recipient-indexable type data.

## 6.1 Message cell

A message cell should have:

### Lock

The sender’s ML-DSA lock.

This ensures the sender retains spending and reclaim authority.

### Type script

A CEMP message type script, or initially a consistent indexing type convention, containing enough information to query messages addressed to a recipient without revealing excessive information.

Recommended type arguments:

```text
version
recipient_route_tag
conversation_tag
message_nonce
```

The exact recipient lock hash should not necessarily be exposed directly. A route tag can be derived:

```text
route_tag = blake2b(
  "CEMP-ROUTE-V1" ||
  recipient_profile_id ||
  routing_epoch
)
```

This still permits traffic analysis, so it should be documented as pseudonymous routing rather than metadata-private messaging.

### Cell data

The cell data should contain a versioned encrypted envelope:

```text
CempEnvelopeV1
 ├── protocol_version
 ├── content_type
 ├── message_id
 ├── conversation_id
 ├── sender_profile_id
 ├── recipient_profile_id or encrypted identity block
 ├── created_at_client
 ├── reply_to
 ├── expiry_hint
 ├── kem_ciphertext
 ├── nonce
 ├── authenticated_header
 └── encrypted_payload
```

Fields that need not be public should be placed inside the encrypted payload.

## 6.2 Encrypted payload

```text
EncryptedMessagePayloadV1
 ├── message_id
 ├── body_type
 ├── text
 ├── attachment_manifests[]
 ├── reply_to_message_id
 ├── reply_to_outpoint
 ├── client_timestamp
 ├── sender_device_id
 ├── receipt_request
 └── padding
```

Authenticated encryption additional data should bind:

* Protocol version
* Network
* Sender profile
* Recipient profile
* Message ID
* Conversation ID
* Payload type
* CKB output identity where available

## 6.3 Conversation identifier

For one-to-one conversations:

```text
conversation_id = blake2b(
  "CEMP-CONVERSATION-V1" ||
  sort(profile_id_A, profile_id_B)
)
```

This provides deterministic conversation association.

A separate routing tag should be rotated periodically if reducing long-term linkability becomes a priority.

# 7. Message lifecycle

## 7.1 Sending

1. User composes a message.
2. App resolves the contact’s current Profile Cell.
3. App verifies the saved profile fingerprint and detects key changes.
4. Message ID and client nonce are generated.
5. Payload is encrypted using the recipient’s ML-KEM key.
6. Transaction builder creates a sender-owned CEMP message cell.
7. The sender signs using ML-DSA.
8. Transaction is broadcast.
9. Local message is stored immediately as `submitting`.
10. Sync worker follows the transaction until committed.
11. Status becomes `available_on_chain`.

## 7.2 Receiving

1. Recipient sync worker queries for CEMP cells matching its active route tags.
2. Unknown outpoints are downloaded.
3. Envelope structure and size limits are validated before decryption.
4. ML-KEM decapsulation and AEAD decryption are attempted.
5. Sender identity and profile state are checked.
6. Message is saved in the encrypted local database.
7. Attachments are queued for download.
8. Message is displayed.
9. A local notification is generated.
10. The app schedules a receipt or includes it in the recipient’s next response.

Reading a message must not spend the message cell.

## 7.3 Response as acknowledgement

A response message includes:

* `reply_to_message_id`
* `reply_to_outpoint`
* Receipt status for the original message
* Optional attachment receipt hashes
* Response body

When the original sender receives this response:

1. The response is decrypted.
2. The referenced outgoing message is found locally.
3. Its state becomes `acknowledged`.
4. Its message cell and related CKBFS cells become eligible for reclaim.
5. Reclaim service batches eligible cells into a transaction.
6. After the reclaim transaction commits, the original message becomes `reclaimed`.

## 7.4 Recipient-side cleanup monitoring

The user who sent the response should monitor the original message outpoint.

State sequence:

```text
received
→ response_sent
→ awaiting_sender_reclaim
→ source_cell_spent
→ local_reclaim_marker_removed
```

When `get_live_cell` or indexer state shows that the initial sender has spent the original cell:

* Mark the remote transport copy as reclaimed.
* Delete the local outpoint-monitor record.
* Remove temporary encrypted transport blobs.
* Retain the locally decrypted message unless the user has enabled disappearing local messages.

“Mark itself for deletion” should apply to the transport-monitoring record and temporary downloaded chain data, not automatically to the user’s permanent conversation history.

## 7.5 No-response fallback

Some messages may never receive a response.

The sender therefore needs:

* Manual reclaim
* Reclaim after recipient read receipt
* Reclaim after configurable age
* Reclaim all eligible cells
* Emergency capacity recovery

A default expiry policy might be:

* Reclaim after explicit acknowledgement: immediately or in next batch
* Reclaim unacknowledged text after 14 days
* Reclaim unacknowledged images after 3–7 days
* Ask before reclaiming messages that may not have been downloaded

Expiry is a local policy hint, not proof that the recipient received the message.

# 8. Receipt model

Receipt statuses:

```text
0x00 unknown
0x01 downloaded
0x02 decrypted
0x03 displayed
0x04 replied
0x05 attachment_downloaded
0x06 rejected
```

For privacy, the default user-facing system should send only:

* `received`
* `replied`

Read receipts should be opt-in.

Receipts can be:

1. Included in the next response message.
2. Batched into a lightweight receipt message.
3. Sent immediately when the user has enabled delivery receipts.

An acknowledgement cell should also remain sender-owned by the acknowledgement sender, making it reclaimable after the other party responds or after expiry.

# 9. CKBFS image transport

## 9.1 Client-side preprocessing

Before encryption or upload:

1. Remove unsupported metadata.
2. Correct orientation.
3. Preserve aspect ratio.
4. Resize to a respectful maximum.
5. Re-encode.
6. Generate a thumbnail.
7. Show the expected CKB capacity and fee before sending.

Suggested initial limits:

* Maximum longest edge: 1280 pixels
* Preferred chat image longest edge: 960 pixels
* Thumbnail longest edge: 320 pixels
* Preferred format: WebP
* JPEG fallback for photographic content
* Maximum compressed image target: 300–500 KB
* Hard initial protocol maximum: 1 MB
* Animated images: unsupported in version 1

The compressor should progressively reduce:

1. Dimensions
2. Quality
3. Chroma detail

It should not distort aspect ratio.

## 9.2 Encryption before CKBFS

Images must be encrypted before their chunks are published.

Process:

```text
resized image
→ random attachment key
→ AEAD encryption
→ encrypted chunking
→ CKBFS cells
→ encrypted manifest in message payload
```

The attachment key is placed only inside the message’s ML-KEM-protected encrypted payload.

CKBFS should never receive plaintext private images.

## 9.3 Attachment manifest

```text
AttachmentManifestV1
 ├── attachment_id
 ├── ckbfs_root
 ├── chunk_outpoints[]
 ├── encrypted_size
 ├── plaintext_size
 ├── mime_type
 ├── width
 ├── height
 ├── thumbnail
 ├── content_hash
 ├── cipher_hash
 ├── encryption_nonce
 ├── encryption_algorithm
 └── reclaim_group_id
```

Where possible, use a compact root/manifest reference rather than listing every outpoint inside the main message.

## 9.4 Receiving images

1. Decrypt message manifest.
2. Validate declared size against limits.
3. Fetch CKBFS chunks.
4. Validate encrypted chunk hashes.
5. Reassemble encrypted image.
6. Decrypt locally.
7. Validate plaintext content hash.
8. Decode in a sandboxed image pipeline.
9. Store encrypted local copy.
10. Render from local storage.
11. Send attachment receipt.
12. Remove raw downloaded chunk cache.

## 9.5 Image reclaiming

The original image sender owns:

* CKBFS data cells
* Manifest cells
* Main message cell

After an attachment receipt or response:

* Add all related cells to one reclaim group.
* Build a batched reclaim transaction.
* Return capacity to the sender’s operational wallet.
* Mark the attachment transport state as reclaimed.

The recipient monitors the manifest or root cell and removes its reclaim-monitor record once spent.

# 12. Background synchronisation

Android background work should use WorkManager.

Workers:

* Profile refresh worker
* Incoming message discovery worker
* Pending transaction worker
* Watched outpoint worker
* Attachment download worker
* Receipt sender worker
* Reclaim batch worker
* Balance refresh worker
* Database maintenance worker

Because Android may delay background jobs, the app should also perform an immediate synchronisation when:

* Opened
* Returned to foreground
* Conversation opened
* User pulls to refresh
* User sends a message
* Network becomes available

Do not promise real-time delivery equivalent to a push-server messenger. Initial delivery latency depends on:

* Block production
* Transaction inclusion
* Indexer synchronisation
* Android background execution

An optional notification relay could eventually report only that new route-tag activity exists, without carrying message content. It must not be required for protocol operation.

# 14. Cryptographic requirements

## 14.1 Version 1 algorithms

* Transaction authorization: ML-DSA-65
* Recipient key encapsulation: ML-KEM-768
* Payload encryption: AES-256-GCM
* KDF: HKDF-SHA-256 or carefully specified BLAKE2b KDF
* Hashing: CKB-compatible BLAKE2b-256 where chain compatibility is required
* Local password derivation: Argon2id preferred; Scrypt acceptable where implementation constraints apply
* Randomness: operating-system CSPRNG only

The current prototype’s “shared secret directly through personalized BLAKE2b” construction should be replaced by a formally specified HKDF construction.

Example:

```text
PRK = HKDF-Extract(
  salt = envelope_nonce,
  IKM = ml_kem_shared_secret
)

message_key = HKDF-Expand(
  PRK,
  "CEMP-MESSAGE-KEY-V1" || sender_id || recipient_id,
  32
)
```

## 14.2 Domain separation

Distinct domain strings must be used for:

* ML-DSA transaction signatures
* Profile signatures
* Contact bundle signatures
* Conversation IDs
* Routing tags
* Message IDs
* Payload encryption keys
* Attachment encryption keys
* Local backup encryption
* Device pairing

## 14.3 Transaction signing gate

Before mainnet, replace the prototype transaction-hash-only signing path with the exact reviewed CKB sighash-all algorithm expected by the deployed lock.

Tests must prove that signing commits to:

* Raw transaction
* Relevant witness groups
* Input ordering
* Output capacities
* Output locks
* Output type scripts
* Output data
* Cell dependencies
* Header dependencies

## 14.4 Algorithm agility

Every serialized object must identify:

* Algorithm family
* Parameter set
* Protocol version
* Serialization version

Do not assume ML-DSA-65 and ML-KEM-768 will remain the only choices.

# 15. Privacy limitations

The application provides payload confidentiality but does not automatically hide blockchain metadata.

Observers may infer:

* That a message-like cell was created
* Cell size
* Transaction timing
* Funding source
* Reclaim timing
* Repeated route-tag activity
* Probable relationships between transaction participants
* Attachment size ranges

The product must not claim to provide Monero-like metadata privacy.

Potential later mitigations:

* Rotating routing tags
* Message padding buckets
* Reclaim batching
* Delayed reclaim windows
* Coin selection designed to reduce linkage
* Multiple derived operational accounts
* Off-chain rendezvous combined with on-chain settlement
* Fiber-based transport modes
* Private RPC access
* Tor support

# 19. Coding-agent operating rules

Give the coding agent these permanent constraints:

```text
1. Do not change protocol serialization without updating the protocol
   specification, golden vectors and migration version.

2. Never log mnemonics, seeds, secret keys, decrypted payloads, vault
   passwords, database keys or plaintext attachments.

3. Never store permanent plaintext messages or images outside the encrypted
   application database and encrypted attachment directory.

4. Treat RPC and indexer responses as hostile input.

5. Every background operation must be idempotent.

6. Every transaction must be journaled locally before broadcast.

7. Never infer successful delivery from transaction commitment alone.

8. Never delete local user history merely because its transport cell was
   reclaimed.

9. The sender must retain reclaim authority over message and attachment cells.

10. Do not introduce a central service as a protocol dependency.

11. Testnet and mainnet configuration must be structurally separate.

12. Do not enable mainnet until the configured ML-DSA lock deployment and
    transaction signing implementation have passed the formal readiness gate.

13. All cryptographic algorithms and serialized objects must be versioned.

14. New native functionality must be exposed through a platform-neutral
    interface so an iOS implementation can be added later.

15. Keep blockchain terminology out of the ordinary chat workflow unless an
    error requires it.
```

# 20. Recommended first coding milestone

The first coding-agent assignment should not attempt the entire mobile app.

Build a headless two-user testnet reference client that proves:

```text
Create Alice from BIP39
Create Bob from BIP39
Create both Profile Cells
Alice discovers Bob
Alice encrypts and publishes a message
Bob discovers and decrypts it
Bob publishes a response referencing Alice’s outpoint
Alice discovers the response
Alice reclaims the original cell
Bob detects the original cell was spent
All balances and local state reconcile
```

Once this lifecycle is deterministic and covered by integration tests, connect it to the Android messenger UI.

That order prevents the most difficult part—the cell ownership, acknowledgement and reclaim state machine—from becoming entangled with mobile interface work.
